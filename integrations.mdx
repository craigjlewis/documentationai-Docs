## Overview

The trading-app-web stack relies on several external services for AI analysis, scraping, PDF generation, storage, async processing, and market data. Correctly wiring these integrations is required for a functional environment.

Most integrations are configured through environment variables, with defaults or fallbacks defined in `api/core/config.py` and `.env.example`.

<Callout kind="info">
Check `.env.example` as the source of truth for required environment variables and their expected formats, then override values in your own `.env`.
</Callout>

## Main integrations

Use these core integrations to run the full pipeline: scraping via Claude Bridge and Playwright, AI analysis with Claude, PDF generation with Documentero, artifact storage in Supabase, async jobs with Redis/Celery, and market data via AlphaVantage and EODHD.

## Optional vs required integrations

Not every integration is required for a basic development setup. Some services are mandatory for the API to start, while others are only needed for specific features like PDF export or additional data enrichment.

<Callout kind="info">
Core runtime services usually required:
- Claude / Anthropic API key (or Claude Bridge configured)
- Redis and Celery broker/result backend
- Supabase URL and keys (if you store artifacts in Supabase)

Feature-specific and optional:
- Documentero and BrowserAct Documentero API key (PDF exports)
- AlphaVantage and EODHD API keys (market data enrichment)
- Playwright / TradingView scraper service (TVS/TradingView scraping)

If you disable an optional integration, also disable or guard any code paths that depend on it.
</Callout>

## Claude and Claude Bridge

Claude / Anthropic powers AI reasoning, and Claude Bridge acts as an HTTP bridge that BrowserAct and other services call. The backend falls back from `CLAUDE_API_KEY` to `ANTHROPIC_API_KEY` when needed.

You must either provide a valid Claude API key directly or run a Claude Bridge service reachable at the configured URL.

### What it is used for

- Running AI analysis over scraped trading content.
- Powering BrowserAct tasks that require long-context reasoning.
- Supporting any internal tools that call Anthropic models.

### What must be running

- Direct Claude integration: no additional process beyond the backend, just a valid API key.
- Claude Bridge integration: a reachable bridge service, often a local Docker container listening on `http://localhost:3009` by default.

### Environment variables

<ParamField body="CLAUDE_API_KEY" param-type="string" required="false" show-location="false">
Primary API key for Anthropic Claude. The app uses this when available to authenticate AI calls.
</ParamField>

<ParamField body="ANTHROPIC_API_KEY" param-type="string" required="false" show-location="false">
Fallback Anthropic API key used when `CLAUDE_API_KEY` is not set. Set at least one of these keys.
</ParamField>

<ParamField body="CLAUDE_BRIDGE_URL" param-type="string" required="false" show-location="false">
Base URL for the Claude Bridge service. Defaults to `http://localhost:3009` when unset. Required if your AI calls go through a bridge instead of hitting Anthropic directly.
</ParamField>

<Callout kind="tip">
In development, start Claude Bridge on `http://localhost:3009` and keep your Anthropic key in the bridge process. This limits direct key exposure in the app.
</Callout>

## Documentero PDF generation

Documentero is used to generate PDF reports from BrowserAct runs and other analysis outputs. The backend calls Documentero’s API with HTML or structured data and receives a PDF in response.

### What it is used for

- Generating PDF artifacts from BrowserAct reports.
- Producing downloadable summaries of trading analyses.

### What must be running

- A reachable Documentero API instance (cloud or self-hosted).
- Network access from the backend to the Documentero endpoint.

### Environment variables

<ParamField body="DOCUMENTERO_API_KEY" param-type="string" required="false" show-location="false">
API key used when contacting the Documentero service to generate PDFs for generic use cases.
</ParamField>

<ParamField body="BROWSERACT_DOCUMENTERO_API_KEY" param-type="string" required="false" show-location="false">
Dedicated Documentero API key used by the BrowserAct pipelines for report PDF generation. Set this when running BrowserAct end-to-end.
</ParamField>

<Callout kind="tip">
If you only need HTML reports in development, you can leave Documentero keys unset and skip PDF generation paths until you are ready to test exports.
</Callout>

## Supabase storage and auth

Supabase provides storage and database capabilities for artifacts produced by BrowserAct and other services. The app uses both the anon key and a service role key for different trust levels.

### What it is used for

- Storing PDFs and other generated report files.
- Persisting metadata and structured outputs for later retrieval.
- Serving assets back to the frontend via signed URLs.

### What must be running

- A Supabase project with:
  - Base URL enabled.
  - Auth and storage configured.
- Network access between the backend and Supabase.

### Environment variables

<ParamField body="SUPABASE_URL" param-type="string" required="true" show-location="false">
Base URL of your Supabase project, typically `https://xyzcompany.supabase.co`. Used for all Supabase API calls.
</ParamField>

<ParamField body="SUPABASE_ANON_KEY" param-type="string" required="true" show-location="false">
Public anon key used by client-side or low-privilege operations. Only grant minimal required scopes.
</ParamField>

<ParamField body="SUPABASE_SERVICE_ROLE_KEY" param-type="string" required="true" show-location="false">
Service role key used by the backend for privileged operations like server-side inserts or storage actions. Keep this secret and only use it on the server.
</ParamField>

<Callout kind="alert">
Never expose `SUPABASE_SERVICE_ROLE_KEY` in frontend code, logs, or client-side configuration. Treat it as a secret with full database access.
</Callout>

## Redis and Celery async processing

Redis and Celery power asynchronous jobs such as scraping, BrowserAct full pipelines, and other long-running tasks. The backend enqueues jobs, and separate Celery workers consume them.

### What it is used for

- Running TVS/TradingView scrapes without blocking HTTP requests.
- Executing BrowserAct pipelines end-to-end (scrape → analyze → store).
- Offloading long-running or scheduled background tasks.

### What must be running

- A Redis instance reachable by both the API and Celery workers.
- One or more Celery worker processes configured with the same broker/result backend as the API.
- Optional Celery beat or scheduler if you rely on periodic tasks.

### Environment variables

<ParamField body="REDIS_URL" param-type="string" required="true" show-location="false">
Connection URL for your Redis instance, for example `redis://localhost:6379/0`. Used by Celery and potentially other caching layers.
</ParamField>

<ParamField body="CELERY_BROKER_URL" param-type="string" required="true" show-location="false">
Broker URL for Celery task transport. Often the same as `REDIS_URL` when using Redis as the broker.
</ParamField>

<ParamField body="CELERY_RESULT_BACKEND" param-type="string" required="true" show-location="false">
Backend URL where Celery stores task results and states. Frequently matches `REDIS_URL` when using Redis as both broker and backend.
</ParamField>

<Callout kind="tip">
In local development, point `REDIS_URL`, `CELERY_BROKER_URL`, and `CELERY_RESULT_BACKEND` to the same Redis instance to keep configuration simple.
</Callout>

## AlphaVantage market data

AlphaVantage provides market and time-series data used by the trading stack. The backend calls AlphaVantage to retrieve prices, indicators, or other metrics that feed into reports and analyses.

### What it is used for

- Fetching historical and intraday price data.
- Pulling technical indicators used in trading strategies.
- Supplementing BrowserAct or other reporting flows with numeric market context.

### What must be running

- No local service is required, but:
  - You must create an AlphaVantage account.
  - The AlphaVantage API must be reachable from the backend host.

### Environment variables

<ParamField body="ALPHA_VANTAGE_API_KEY" param-type="string" required="false" show-location="false">
API key used to authenticate requests to AlphaVantage. Required for any feature that fetches data from AlphaVantage endpoints.
</ParamField>

<Callout kind="info">
If you do not set `ALPHA_VANTAGE_API_KEY`, any feature depending on AlphaVantage market data will fail or need to be disabled. Check the code paths where AlphaVantage is imported before leaving this unset.
</Callout>

## EODHD optional market data enrichment

EODHD (End-of-Day Historical Data) is used as an optional data source for BrowserAct’s full pipeline. When configured, it can enrich reports with additional historical data.

### What it is used for

- Enriching BrowserAct reports with extended end-of-day historical data.
- Providing an alternative or complementary feed to AlphaVantage.

### What must be running

- An active EODHD subscription or API access.
- Internet access from the backend to EODHD endpoints.

### Environment variables

<ParamField body="EODHD_API_KEY" param-type="string" required="false" show-location="false">
API key used when calling EODHD endpoints for additional market data. The exact variable name may differ slightly depending on the repo version; confirm in `.env.example` and align your configuration accordingly.
</ParamField>

<Callout kind="info">
Repo docs mention EODHD integration for BrowserAct enrichment. If your `.env.example` uses a slightly different name for this key, follow that name instead of `EODHD_API_KEY`.
</Callout>

## Playwright and TradingView scraper

A Playwright-based scraper is used to pull content from TVS/TradingView and similar sources. It runs as part of the services layer, typically coordinated by Celery tasks.

### What it is used for

- Scraping TradingView and related UIs for charts, watchlists, or sentiment.
- Feeding raw HTML or structured data into BrowserAct and Claude for analysis.

### What must be running

- Playwright dependencies installed (usually via `playwright install` and browser downloads).
- A running service or worker process that executes `tvs_scraper.py` (or the equivalent module under `api/services/tvs/tvs_scraper.py`).
- Redis/Celery configured if scrapes are triggered asynchronously from API requests.

### Environment variables

Playwright and the TradingView scraper primarily use existing infrastructure configuration (Redis, Celery, Claude/BrowserAct, Supabase) rather than dedicated env vars. Ensure the following are correctly set so scrapes can run end-to-end:

<ParamField body="REDIS_URL" param-type="string" required="true" show-location="false">
Shared Redis instance used when scraper tasks are enqueued or coordinated via Celery.
</ParamField>

<ParamField body="CELERY_BROKER_URL" param-type="string" required="true" show-location="false">
Broker URL that scraper tasks use to communicate with Celery workers.
</ParamField>

<ParamField body="CELERY_RESULT_BACKEND" param-type="string" required="true" show-location="false">
Result backend used to track the status and outputs of scraping tasks.
</ParamField>

<Callout kind="tip">
If scraping fails silently, first verify that Celery workers are running with the same `CELERY_BROKER_URL` and `CELERY_RESULT_BACKEND` as the API, then check Playwright installation by running a simple headless browser script locally.
</Callout>
