## What is trading-app-web?

trading-app-web is a full-stack trading application monorepo that combines a FastAPI backend, multiple user interfaces, background task processing, and FX reporting pipelines.

Use this project as the central entry point for running the trading API, web UI, legacy Streamlit dashboards, an optional Electron desktop wrapper, and the BrowserAct FX report workflow.

## High-level architecture

The repository groups several apps and services under a single codebase:

- A FastAPI backend that exposes REST endpoints under `api/main.py` with routers mounted under `/api/v1`.
- A React/Vite frontend stored under `frontend/src` that consumes the backend API.
- A Streamlit app (`app.py` plus `pages/`) used as a legacy or alternative UI.
- An Electron desktop app (`electron/main.js`) that wraps the web frontend for desktop usage.
- Celery workers (`celery_app.py` and `tasks/`) for running asynchronous and scheduled jobs.
- Database models and migrations using SQLAlchemy and Alembic (`database/`, `migrations/`).
- A Claude Bridge service under `claude-bridge/` for AI-assisted workflows.
- A BrowserAct FX report pipeline that spans frontend and backend for FX reporting.

## Backend API

The backend API is a FastAPI application that exposes core trading, reporting, and support endpoints.

Key characteristics:

- Application entry point at `api/main.py`.
- Routers grouped under `api/v1` for versioned endpoints.
- Database access via SQLAlchemy models in the `database/` directory.
- Schema changes managed via Alembic migrations in `migrations/`.

<Callout kind="tip">
  When you start exploring the code, begin with `api/main.py` and the routers under `api/v1` to understand the primary endpoints and how they are wired.
</Callout>

## Frontend UI

The primary web UI is a React application bundled with Vite.

- Source code lives under `frontend/src`.
- Pages and views map onto backend endpoints for trading and reporting flows.
- The BrowserAct FX report page is implemented in `frontend/src/pages/BrowserAct.tsx` and communicates with the matching backend routes.

Use this UI as the default interface for web-based trading workflows.

## Streamlit app

The Streamlit app provides a legacy or alternative interface for parts of the trading system.

- Main entry point: `app.py`.
- Additional pages and flows live under the `pages/` directory.
- Useful for dashboards, exploratory analysis, or workflows that benefit from Streamlit’s rapid layout primitives.

Treat this as an optional UI layer; the React/Vite frontend is the primary user interface.

## Electron desktop app

The Electron app wraps the web frontend in a desktop shell.

- Entry point: `electron/main.js`.
- Uses the existing frontend build to display the UI in a desktop window.
- Suitable when you need a desktop-distributed version of the trading client but want to reuse the same web stack.

## BrowserAct FX reports

BrowserAct FX reports provide a specialized FX reporting pipeline that spans frontend, backend, and AI services.

Key pieces:

- **Frontend page**: `frontend/src/pages/BrowserAct.tsx` implements the BrowserAct reporting interface.
- **API routes**: FX-related endpoints are defined in `api/routes/browseract_reports.py`.
- **Services**: Supporting logic for transformer, AI, and Documentero integration is implemented in dedicated service modules.

This pipeline lets you ingest browser activity or FX data, process it via AI services, and generate structured FX reports.

## Automation and tasks

Background processing and automation rely on Celery and database tooling.

- **Celery configuration**: `celery_app.py` defines the Celery application.
- **Tasks**: The `tasks/` directory contains Celery task definitions for asynchronous workloads such as report generation or data ingestion.
- **Database**: SQLAlchemy models and session management live under `database/`.
- **Migrations**: Alembic migration scripts live in `migrations/` to track schema evolution.

Run these services alongside the API to handle long-running or scheduled jobs reliably.

## Claude Bridge service

The Claude Bridge service integrates AI capabilities into the trading stack.

- Located under `claude-bridge/`.
- Acts as a bridge between the core application and Claude-based AI features.
- Typically used by backend services or FX/reporting workflows to enrich or analyze data.

Review this service when you need to adjust AI-related behavior, prompts, or integration patterns.

## Repository layout

Use this summary when you are navigating the codebase for the first time:

- `api/` – FastAPI application modules, including `main.py` and routers under `v1`.
- `frontend/` – React/Vite web application source under `frontend/src`.
- `pages/` – Streamlit page modules used by `app.py`.
- `tasks/` – Celery task definitions for background and scheduled jobs.
- `migrations/` – Alembic migration scripts for database schema changes.
- `claude-bridge/` – Claude Bridge service code for AI-assisted operations.

## Where to start

If you are new to the repository, a common path is:

- Skim `api/main.py` and the routers to understand available endpoints.
- Open `frontend/src` and the main routing configuration to see how the UI maps to API calls.
- Review `frontend/src/pages/BrowserAct.tsx` and `api/routes/browseract_reports.py` if you work specifically on FX reports.
- Explore `celery_app.py`, `tasks/`, and `migrations/` when you need to touch background jobs or database schema.
